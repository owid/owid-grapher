Writing at OWID, natively

A rich, in-browser editing experience for our content — replacing ArchieML-in-Google-Docs with a block editor that knows our components, previews live, and hosts feedback where the writing happens.

What this costs us

  • Authors write syntax, not documents — {.chart} blocks, key–value props, indentation rules.
  • The feedback loop is edit → switch tab → wait for refetch → full iframe reload, losing scroll position.
  • Components are invisible until previewed; discovering what exists means reading docs.

What Google Docs gives us today (must not regress)

  • Fluent prose writing — typing is never interrupted.
  • In-situ comments & suggestions — our whole review culture lives there.
  • Revision history — safe to experiment, easy to restore.
Codebase investigation

The foundations already exist

Content model

The block JSON is already the CMS

ArchieML is only an ingestion syntax. Rendering, baking and search all consume OwidEnrichedGdocBlock[] stored in posts_gdocs.content — a typed union of ~65 block types. The editor operates on this model directly; ArchieML disappears.

Rendering

Every block renders client-side

ArticleBlock.tsx dispatches all block types in the browser. The SSR iframe preview is a historical choice, not a necessity — the site renderer can be the live editing canvas itself.

Charts

The chart editor is a reusable engine

Grapher renders from a config, fully client-side. The editor is factored as AbstractChartEditor + tab components + a live <Grapher>, already reused three ways — embeddable as an in-article overlay.

Greenfield

Comments & collaboration: nothing exists

All review workflow lives inside Google Docs. Comments, presence, and revision history are the genuinely new subsystems we take ownership of. Images, by contrast, are already fully native (Cloudflare Images + picker UI).

The canvas is the site renderer — the same React components the public page uses, wrapped in editing chrome. No iframe, no refetch, no lost scroll position.

Principle 1 · Fluency

Writing must never wait for the tool

  • Type-first. Enter continues a paragraph. Markdown shortcuts as you type: # heading, - list, > quote, **bold**.
  • Slash menu. / opens an inline inserter, fuzzy-searchable, ranked by real per-doc-type usage — a heading is two keystrokes, never a drag.
  • Drag & drop is for arranging, not for writing. The palette exists for discovery; the keyboard is the fast path.
  • Paste from Google Docs works. Headings, lists, links, bold — pasted prose becomes proper blocks. This is also our manual migration story.
  • Undo is universal. One history across text edits, block moves, and settings.

Principle 2 · Components as first-class citizens

Every block renders real, and knows how to be edited

Three kinds of blocks, three editing modes

  • Flow blocks (text, headings, lists, quotes, callouts) — edited directly in place, like any document editor.
  • Atom blocks (chart, image, video, prominent-link, cta …) — render the actual site component, selected as a unit. Hover reveals a toolbar: cogwheel → inspector, drag, duplicate, delete.
  • Containers (sticky-right, side-by-side, gray-section, expander) — real layout CSS in the canvas, with drop zones for their children.

And a safety valve

  • Any block type the editor doesn't fully support yet renders as a read-only raw card and survives saving untouched — exotic blocks never make a document uneditable.

Principle 3 · The cogwheel

Tweak the chart where you're writing about it

  • Stage A — embed options. Cogwheel opens the block inspector: size, height, tab, query params, caption.
  • Stage B — “Customize this chart.” Overlays a focused slice of the chart editor (basic · text · customize tabs) around the live chart, in the writing surface.
  • The subtlety: charts are shared entities — editing one here would change it on its data page and every other article. Our block model already has the answer: in-situ customization creates a narrative chart (an article-scoped variant), via the flow that exists today.

Feasible because the chart editor is already an embeddable engine — the same machinery powers chart, indicator, and narrative-chart editing today.

Principle 4 · Feedback where the writing is

Comments and review, in-situ

  • Range comments on selected text — anchored as editor marks, they move with the text through edits, splits, and rewrites.
  • Block comments on charts, images, and other atoms — every block gets a stable id (which later also serves AI edits and deep links).
  • Never lost: if commented text is deleted, the thread becomes orphaned and stays listed — same guarantee Google Docs gives.
  • Presence & protection: “Fiona is editing” banners plus stale-write detection. No hard locks; real-time co-editing (Yjs) is architected-for but deferred — it will matter for agent collaboration too.
  • Revision history with restore — autosaves pruned, every publish kept forever.

Stage 2 · Designed-for now, built later

An AI co-author with structural hands

  • A chat rail beside the document. “Tighten the intro”, “move the chart above the fold”, “add a callout summarizing the 3 key numbers”.
  • Structural edit tools, not text dumps. The AI edits through the same command layer as the keyboard and palette: insertBlock, replaceBlock, moveBlock, editTextRange — addressed by block ids.
  • Suggestions, not silent edits. AI changes land as reviewable track-changes, meeting the comments system where humans already review.

Why it's cheap later: every editor mutation is already a transaction, blocks already have ids, and the enriched JSON is already the model-friendly representation of a document. Stage 1 builds the rails; stage 2 puts the agent on them.

Direction

Four decisions, made

Source of truth

Native-first, per-doc opt-in migration

New docs are born in the CMS. Existing gdocs keep working unchanged; each converts one-way when its authors are ready. No write-back to Google. The two pipelines coexist until adoption retires the old one, doc type by doc type.

Editor foundation

TipTap / ProseMirror

The industry-standard engine behind Notion-class editors. React node views let our blocks render through the existing site components; the richest ecosystem for comments, track-changes, and (later) Yjs collaboration.

Collaboration v1

Comments + presence, multiplayer-ready

In-situ threads and soft locking from day one. Real-time co-editing deferred — but the document format and mutation model are chosen to be Yjs-compatible, for humans and agents alike.

First milestone

Data insights first

435 published docs, three block types, constant authoring volume. Validates the entire stack — editor, comments, publishing — in the smallest real arena. Articles follow.

Untouched

  • Baker, public site, search indexing, derived tables — they keep reading the same JSON they read today.
  • The block type system — the editor maps ProseMirror ⇄ OwidEnrichedGdocBlock[] losslessly, proven by a round-trip test over every published document in CI.

New

  • Draft + revision + comment tables; save/publish/presence endpoints.
  • Batched reference resolution (charts, images, indicators) so the canvas renders live — also fixes a known N+1 the current preview suffers from.
  • The editor app itself, in the admin, behind a feature flag.

Risks & how we hold them

RiskMitigation
Serialization infidelity corrupts content Round-trip harness over the full published corpus in CI from day one; unsupported blocks pass through opaquely; publish snapshots are immutable and restorable.
Editing feels worse than Google Docs → no adoption Data insights first (small blast radius); paste-from-gdocs quality bar; revision history at parity before pushing conversion; authors in the beta from M1.
Canvas performance with many live charts Charts render as thumbnails until visible/focused, then hydrate; measured against our heaviest real articles before M2 exits.
Two authoring pipelines forever One-way conversion (the native set only grows) + per-doc-type “default to native” switches + an explicit retirement criterion per doc type.
Site CSS vs admin CSS conflicts in the canvas Scoped canvas stylesheet, spiked in M0 before anything depends on it.
