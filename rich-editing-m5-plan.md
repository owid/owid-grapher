# Rich editor M5 ŌĆö Real-time collaboration & remaining doc types

_Drafted 2026-07-04. Companion to `rich-editing-technical-plan.md` (┬¦4.3, ┬¦10). Sequenced **before** the M4 AI side panel per Daniel's decision: the sync server, live cursors, and reconciliation are the substrate agent collaboration builds on ŌĆö an agent should join a document the way a colleague does (as a Yjs client with awareness presence), not through a bolted-on request/response loop._

## 0. Goals & non-goals

**Goals**

1. **Yjs sync server**: multiple people (and later agents) edit the same draft concurrently; the 409 "someone saved a newer version" conflict path disappears for connected clients.
2. **Live cursors & presence**: remote selections and carets in the canvas; awareness replaces the in-memory heartbeat presence (fixing the per-process debt).
3. **Document reconciliation**: a defined, tested story for every place the CRDT world meets the JSON world ŌĆö seeding, materializing, restoring, schema migration, ydoc disposal.
4. **Stable block identity**: blocks get durable ids, which comments (block threads), restore, and the M4 command layer all need.
5. **Remaining doc types**: topic pages, linear topic pages, about pages, authors, announcements, profiles, fragments, homepage become natively editable.

**Non-goals (deferred to M4)**

- The chat rail, Claude API proxy, suggestion/track-changes mode. But ┬¦6 defines the *interfaces* M4 consumes (SelectionRef, agent awareness, block addressing) so nothing here has to be reworked.

**Invariant carried through everything below**: enriched JSON remains the durable format. The ydoc is a live-collaboration transport ŌĆö always discardable, always re-seedable from JSON. History, restore, publish, bake never depend on CRDT internals.

---

## 1. Block identity (do this first ŌĆö everything else touches it)

Today blocks have no ids: the inspector addresses the `NodeSelection` position, chart sessions remap positions through transactions, comments use marks. That works for one user's live session but not for durable references (block comment threads, agent commands like "rewrite block X", anchoring across a ydoc reseed).

**Design**

- Every top-level-block PM node (atoms *and* containers, the same set that has a `BlockFrame`) gets a `blockId` attr: nanoid(10), generated by a plugin on node creation, de-duplicated on paste/split (a paste carries the source's id ŌåÆ regenerate when the id already exists in the doc).
- **Persisted in draft JSON**: `pmToEnriched` emits it as an optional `id?: string` on `OwidEnrichedGdocBlock`; `enrichedToPm` reads it back. This is required ŌĆö the ydoc is disposable, so ids must survive a JSONŌåÆydoc reseed or block-anchored comments and agent refs would break.
- **Stripped at publish**: the publish snapshot into `posts_gdocs.content` removes ids (a `stripBlockIds` pass), so published content, the baker, search, and the round-trip corpus are untouched. Revisions of kind `publish` are stored stripped; `autosave`/`manual`/`restore` revisions keep ids (they're drafts).
- Corpus round-trip stays green by construction: published docs have no ids, and `normalizeForComparison` additionally ignores `id` so re-converted docs compare clean.
- Nested content (a paragraph inside a sticky-right column) is addressed as `blockId` of the container + a path within it; we do NOT give ids to every leaf. If M4 needs finer targets, text ranges inside a block are addressed with relative positions (┬¦6).

**Sizing**: serialization emit/read + strip pass + id plugin + tests. Small, but it must land before the sync server ships so we never have a ydoc generation without ids.

## 2. Sync server

**Library**: Hocuspocus (the maintained y-websocket successor: hooks for auth/load/store, awareness built in, Redis extension exists if we ever need multi-process). Mounted on the existing admin Express server via websocket upgrade at `wss://ŌĆ”/admin/api/gdocs/:id/sync` ŌĆö same origin, same session-cookie auth (`onAuthenticate` validates the admin session and loads the user, exactly like `authentication.ts` does for REST).

**Why in-process with the admin server**: it matches current reality (presence is already per-process and fine because the admin runs single-process), avoids new deploy surface on staging (pm2 config stays), and the Redis extension is the documented escape hatch if the admin ever scales horizontally. Sticky infrastructure decisions stay small.

**Document lifecycle**

- `onLoadDocument`: if `posts_gdocs_ydocs` has a blob for this gdoc and its `schemaVersion` matches, load it. Otherwise **seed**: read the draft head JSON (`posts_gdocs_drafts.content`), run `enrichedToPm`, build the PM JSON ŌåÆ `Y.XmlFragment` (via `prosemirrorJSONToYDoc` from y-prosemirror), store with the current `schemaVersion` and the draft `revisionId` it was seeded from.
- `onStoreDocument` (debounced, Hocuspocus default ~2s): two writes in one transaction ŌĆö
    1. the encoded ydoc state (`Y.encodeStateAsUpdate`, which is the compacted form) into `posts_gdocs_ydocs`,
    2. **materialize**: ydoc ŌåÆ PM JSON ŌåÆ `pmToEnriched` ŌåÆ the existing draft-save path (`posts_gdocs_drafts` head + throttled `autosave` revision + pruning + derived-content update for unpublished docs). The REST autosave endpoint's body becomes a shared function; the sync server calls it with the materialized content.
- `onDisconnect` (last client): final store, then optionally **discard** the ydoc row (see ┬¦4 disposal policy).

This is why serialization portability matters: `serialization/` operates on plain PM JSON with no TipTap runtime imports, so it moves from `adminSiteClient/richEditor/serialization/` to `adminShared/richEditor/serialization/` (adminShared is already imported from both sides) essentially as a `git mv`. The PM schema construction needs a server-usable variant ŌĆö the schema shape (`pmJson.ts` node/mark names + attrs) is already shared; the TipTap extension objects stay client-only and we build a plain `prosemirror-model` Schema on the server from the same definitions. This "one schema, two constructions" needs a test asserting they accept the same documents.

**Client**: `RichEditorPage` swaps its autosave/409 machinery for `@hocuspocus/provider` + TipTap's `Collaboration` extension (y-prosemirror under the hood) when the doc is sync-enabled. PM history is replaced by Yjs undo (`Y.UndoManager` scoped to the local client's edits ŌĆö you undo your changes, not your colleague's). The save-status widget becomes a sync-status widget (connected / syncing / offline with local buffering ŌĆö the provider queues updates through disconnects and merges on reconnect, which is the CRDT paying rent).

**Rollout switch**: per-doc-type or per-doc flag (`syncEnabled` derived from a feature flag to start). The REST body-save path stays intact for non-synced docs during rollout and as the fallback if we have to turn the sync server off; it 409s a stale client exactly as today. Retire it once sync has been default for all native docs for a while.

**Storage**

```sql
CREATE TABLE posts_gdocs_ydocs (
    gdocId VARCHAR(255) PRIMARY KEY,          -- FK posts_gdocs.id
    ydoc LONGBLOB NOT NULL,                    -- Y.encodeStateAsUpdate output
    schemaVersion INT NOT NULL,                -- PM schema generation (┬¦4)
    seededFromRevisionId BIGINT UNSIGNED NULL, -- provenance, debugging
    updatedAt DATETIME NOT NULL
)
```

One row per doc, overwritten on every debounced store ŌĆö **not** an append-only update log. `encodeStateAsUpdate` already merges history and GC'd tombstones, so the blob is roughly document-size plus bounded CRDT metadata; the durable fine-grained history continues to live in `posts_gdocs_revisions` as JSON, where it already has pruning semantics. (If we ever observe lost-update pain between debounced stores, the fix is Hocuspocus's incremental persistence ŌĆö append updates, compact into the snapshot on load ŌĆö as a drop-in change to this table plus a log table. Start simple.)

## 3. Live cursors & presence

- y-prosemirror's `yCursorPlugin` (bundled in TipTap's `CollaborationCursor`) renders remote selections: carets with name flags for text selections, and we extend `BlockFrame` to render a colored outline + name chip when a remote user's selection is a `NodeSelection` of that block (awareness gives us the remote selection as relative positions; we decode and check the node span). Users get stable colors hashed from user id.
- **Presence via awareness**: each client sets `{user: {id, fullName, color}}` in the awareness state. The presence banner reads awareness instead of the heartbeat endpoint. The in-memory `presenceByGdocId` map, `PRESENCE_TTL_MS`, the heartbeat endpoint, and the client polling loop are deleted ŌĆö the per-process-presence debt closes as a side effect (awareness lives in the sync connection, which is inherently per-document-server).
- Awareness is also where **agents will appear** in M4: an agent client sets `{user: {id: "agent:ŌĆ”", fullName: "Claude", kind: "agent"}}` and its "cursor" (the block it's working on) renders in the canvas like a colleague's. Nothing to build now beyond keeping the awareness schema open (a `kind` field).

## 4. Document reconciliation

Every CRDTŌåöJSON boundary, made explicit:

| Event | Behavior |
| --- | --- |
| **First open (no ydoc)** | Seed from draft head JSON (┬¦2). Block ids come along because they're in the JSON. |
| **Restore a revision** | Restore must work while others are connected. The restore endpoint no longer writes the draft row directly for synced docs; instead the server opens the ydoc and applies "replace root fragment content with `enrichedToPm(revision)`" **as a Yjs transaction**, which propagates to all clients like any remote edit (and is undoable via their UndoManagers as a remote change is not ŌĆö so the client shows a "document was restored by X" toast via a broadcast, not silent replacement). Materialization then writes the draft head + a `restore` revision as today. |
| **Schema migration** (we add/change PM node types) | `schemaVersion` in `posts_gdocs_ydocs` bumps with any breaking schema change. On load with a stale version: **discard and reseed** from the draft head JSON. Safe because materialization keeps the JSON current to within the debounce window, and cheap because reseeding is exactly the first-open path. This is the disposability invariant earning its keep ŌĆö no ydoc data migrations, ever. |
| **Ydoc disposal** | A nightly job (and the stale-schema path) deletes ydoc rows whose doc has had no connection for N days (start: 7). Reopen = reseed. Bounds table growth structurally rather than by compaction heroics. |
| **Convert-to-native / new doc** | Unchanged ŌĆö they create draft JSON; the ydoc is born lazily on first editor open. |
| **Publish** | Unchanged ŌĆö reads the draft head (materialized JSON), strips block ids, snapshots to `posts_gdocs.content`, existing bake tail. Publishing during a live session is fine: it reads the latest materialization. |
| **Offline / stale client reconnects** | Yjs merges the buffered local updates with the server state; no 409, no dialog. The one sharp edge: a client that stayed offline across a **discard+reseed** (schema bump or disposal) holds updates against a dead ydoc generation. Guard: the provider handshake includes a ydoc generation id (new column or derive from `seededFromRevisionId`); on mismatch the client drops local buffered updates and reloads, telling the user. Rare, explicit, safe. |
| **Comment anchors** | Comment marks live in the doc, so they sync to all clients automatically and materialize into draft JSON as today. The anchor-refresh that ran on save now runs server-side at materialization time. Block threads anchor to `blockId` (finally possible), which survives reseeds; range threads keep mark anchoring. |

Testing this section is most of its cost: a `devTools` reconciliation harness that scripts two headless providers + server through each row of this table (concurrent edit merge, restore-under-concurrency, reseed-after-schema-bump, offline-across-disposal) and asserts the materialized JSON. This is also the regression bed the M4 agent client will run against.

## 5. Remaining doc types

Currently native-capable: `data-insight` (M1), `article` (M2). Remaining, in proposed order:

1. **`linear-topic-page`** ŌĆö article-shaped (same block palette + a few extras); mostly a doc-type schema + settings-form pass. Cheapest, do it while the sync work is in review.
2. **`topic-page`** ŌĆö heavier block tail: `key-insights`, `explorer-tiles`, `research-and-writing` variants, `all-charts` with tag scope; needs the `relatedCharts` attachment scope (doc-level, from tags) added to `resolveReferences` scopes.
3. **`about-page`**, **`announcement`** ŌĆö small palettes, low risk.
4. **`author`**, **`profile`** ŌĆö settings-heavy (bio fields, socials / profile scope), small bodies.
5. **`fragment`** ŌĆö resolves open question #3 from the main plan: DoD dictionary and FAQs get a constrained editor surface (fragments are keyed lists of blocks; the canvas edits one key's block list, a rail list switches keys). DoD editing matters for agents too (agents will want to read DoD definitions as context).
6. **`homepage`** ŌĆö last; singleton, most bespoke blocks, needs the `homepageMetadata`/`latestDataInsights` scopes. Low urgency (edited rarely).

Per-type recurring checklist: doc-type block palette (registry `docTypes` filtering exists), settings rail form (shared mechanics from M1), attachment scopes, publish path reusing the existing tail, conversion report allowances, round-trip corpus slice green for that type, one real doc converted and edited by an author.

The corpus harness already covers all published docs of every type (1,173/1,173), so serialization is largely done; the work is palettes, settings forms, scopes, and per-type publish validation ŌĆö plus first-classing the blocks that only these types use (`key-insights`, `explorer-tiles`, `pill-row`, `homepage-*`; the rest stay raw-passthrough until demanded).

## 6. Agent-readiness (interfaces M4 will consume ŌĆö defined now, built mostly in M4)

- **SelectionRef** ŌĆö the serializable "this" for *do X with this*:

```ts
type SelectionRef =
    | { kind: "block"; blockId: string; blockType: string }
    | {
          kind: "text"
          // y-prosemirror relative positions, JSON-encoded: survive concurrent
          // edits and map to absolute positions in any ydoc generation's view
          anchor: RelPosJson
          head: RelPosJson
          blockId: string // containing top-level block, for context/fallback
          excerpt: string // captured at grab time, for the prompt & orphan UI
      }
    | { kind: "document" }
```

  Built by a small `selectionRefFromEditor(editor)` on top of the existing trio (`selectionBlockKey` / `hasTextRangeSelection` / NodeSelection inspection). Resolved back via `relativePositionToAbsolutePosition` / blockId scan; a ref whose target vanished resolves to null and the consumer shows an orphan state (same philosophy as comment orphans). Comments' block threads should adopt `SelectionRef` semantics so there's one anchoring vocabulary.
- **Agents are Yjs clients**: the M4 proxy edits through a headless provider on the sync server, so agent edits stream into the canvas live, are attributed via awareness (`kind: "agent"`), interleave safely with human typing, and are undoable per-client. The reconciliation harness (┬¦4) doubles as the agent-client test bed.
- Deliver in M5: `SelectionRef` type + capture/resolve + comments migrated to it. Defer to M4: the command layer, prompts, suggestion mode.

## 7. Milestone plan & exit criteria

Suggested PR slicing (each independently green):

1. **M5a ŌĆö Block identity**: id attr plugin, serialization emit/read, publish strip, comments block threads UI (closes the M1 debt) anchored by blockId.
2. **M5b ŌĆö Sync server**: serialization move to adminShared, server schema construction, Hocuspocus mount + auth, ydocs table, seed/store/materialize, client Collaboration swap behind flag, sync-status UI.
3. **M5c ŌĆö Cursors, presence, reconciliation**: CollaborationCursor + BlockFrame remote outlines, awareness presence (delete heartbeat), restore-as-transaction, schema-version reseed, disposal job, generation guard, reconciliation harness.
4. **M5d ŌĆö Doc types**: in the ┬¦5 order; can proceed in parallel with b/c.
5. **M5e ŌĆö SelectionRef** + capture/resolve + comments unification.

**Exit criteria**

- Two authors edit the same article simultaneously in real browsers: both see each other's cursors and block selections, no 409 dialogs, materialized draft/revisions correct, restore works mid-session.
- Kill the sync server mid-edit ŌåÆ clients buffer ŌåÆ restart ŌåÆ merged, nothing lost beyond the debounce window.
- Delete a doc's ydoc row ŌåÆ reopen ŌåÆ identical document (ids included).
- Corpus round-trip still 100% including id round-trip on drafts; unit + db suites green.
- At least one real doc of each of linear-topic-page and topic-page authored/edited natively.
- A scripted headless client can connect, resolve a SelectionRef, and apply an edit that appears live in a browser ŌĆö the M4 smoke test, proven before M4 starts.

## 8. Risks

| Risk | Mitigation |
| --- | --- |
| Ydoc blob growth on hot docs | `encodeStateAsUpdate` compaction on every store + disposal-after-idle; JSON history lives outside the CRDT so disposal is always safe. |
| Two schema constructions drift (client TipTap vs server prosemirror-model) | Shared node/mark definitions in `pmJson.ts`; equivalence test in CI feeding the corpus through both. |
| Materialization bug corrupts drafts silently | Materialization runs `pmToEnriched` + the same validation as the REST save; on failure, store the ydoc but keep the previous draft head and alert (Sentry) ŌĆö never write invalid JSON. |
| Yjs undo UX surprises (undo-mine-only) | It's the standard collab semantic (gdocs does the same); verify chart-session and block-conversion undo paths against UndoManager explicitly. |
| Admin server restart drops live sessions | Providers reconnect + resend buffered updates automatically; debounced store bounds loss; verify in exit criteria. |
| In-process sync server blocks the event loop under load | Materialization is the only heavy sync-path work; it's debounced per-doc and the doc count in simultaneous edit is small (team of ~30). Measure; Redis/worker split is the documented escape hatch. |
